WHAT WAS ACCOMPLISHED

✅ Simplified Role Schema

   - Removed unnecessary complexity (verified, active booleans)
   - Clear status enum (active, pending, blocked)
   - Business-aligned structure

   - Admin role excluded from user-selectable roles
   - Pre-save hook validates no admin assignment

   - Backend-only (internal use protected)

   - Role parameter removed from registration API
   - Simpler onboarding for low digital literacy users
   - Aligns with OmishGo's farmer-centric model

✅ Automatic Migration
   - Old users auto-initialize on first save
   - Pre-save hook detects and handles
   - Zero downtime transition
   - No user action required

✅ 100% Backward Compatible
   - Old role field preserved
   - JWT unchanged
   - Auth flow unchanged
   - Mobile app unaffected

ROLE STRUCTURE BEFORE & AFTER

BEFORE (Enterprise Complex):
┌─────────────────────────────────┐
│ type: String                    │
│ verified: Boolean (def: false)  │
│ subscription: String (def: free)│
│ active: Boolean (def: true)     │
└─────────────────────────────────┘

AFTER (MVP Simple):
┌─────────────────────────────────┐
│ type: String                    │
│ status: Enum (active/pending/   │
│         blocked, def: active)   │
│ subscriptionTier: String        │
│         (def: free)             │
└─────────────────────────────────┘

Result: Cleaner, simpler, more meaningful ✅

REGISTRATION COMPARISON

OLD (Role Selection):
POST /api/v1/auth/register
{
  "name": "John",
  "email": "john@example.com",
  "phone": "1234567890",
  "pin": "1234",
  "role": "farmer"  ← User had to choose
}

NEW (Farmer-First):
POST /api/v1/auth/register
{
  "name": "John",
  "email": "john@example.com",
  "phone": "1234567890",
  "pin": "1234"
  // Role not in request - automatically FARMER
}

Benefit: Simpler, faster, aligns with business model ✅
