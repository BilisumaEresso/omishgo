# POST MVP BACKLOG

- The roles/subscription module (path: src/modules/roles/) — status: built but not wired to the User schema (references user.activeRole and user.roles which don't exist on User model). To re-enable: add activeRole and roles[] fields to the User schema, then re-mount the router in app.js.
- The device-lock module (path: src/utils/device.service.js) — status: built but never called from auth.service.js login flow. To re-enable: call validateDeviceForLogin() and updateUserDevice() inside loginUser() in src/modules/auth/auth.service.js.
- Both are deferred until after the MVP pilot per the MVP doc's scope section.
- The product-approval admin endpoints (`GET/PUT /api/v1/admin/products*`, i.e. `getPendingProducts`/`approveProduct`/`rejectProduct` in `admin.controller.js`) are broken, not just unused: they read/write product `status` values `"pending"`, `"approved"`, `"rejected"`, none of which exist in the `Product` model's actual enum (`active│sold│draft`). `getPendingProducts` will always return an empty list, and `approveProduct`/`rejectProduct` will throw a Mongoose validation error if ever called. The MVP doc's scope has no product-approval step at all — only user/account approval by Admin — so this looks like leftover scaffolding for a feature that was never aligned with the schema. Left as-is rather than silently fixed/removed, since "should products need admin approval before going live" is a product decision, not a hygiene one. To fix: either add `pending/approved/rejected` to the Product status enum and wire `createProduct` to default new listings to `pending`, or remove these three admin endpoints/routes if product approval isn't wanted.
